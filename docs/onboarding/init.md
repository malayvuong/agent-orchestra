# agent-orchestra init

The `init` command sets up Agent Orchestra in an existing project. It auto-detects available providers (CLI tools and API keys), generates provider configuration, and creates an `AGENTS.md` file for AI coding agents.

Install the CLI first if you have not already:

```bash
npm install -g @malayvuong/agent-orchestra
```

## Usage

```bash
# Basic initialization — auto-detects everything
ao init

# With policy and custom skillset scaffolding
ao init --with-policy --with-skillsets

# Override detected project type
ao init --project-type python

# Force overwrite existing generated files
ao init --force

# Refresh only the detected provider defaults in agents.yaml
ao init --refresh-agents

# Initialize a specific directory
ao init --path /path/to/project
```

## What it does

1. **Detects project type** from filesystem signals (package.json, Cargo.toml, pyproject.toml, etc.)
2. **Detects available providers** — scans for CLI tools on PATH and API keys in environment
3. **Generates `.agent-orchestra/agents.yaml`** with the best available providers assigned to architect and reviewer
4. **Generates `AGENTS.md`** with practical Agent Orchestra instructions tailored to the project
5. **Creates `.agent-orchestra/`** directory structure
6. **Bootstraps built-in review skills** into `.agent-orchestra/skills/` and `.agent-orchestra/skillsets.builtin.yaml`
7. **Optionally generates** `policy.yaml` and a starter custom `skillsets.yaml`

## Provider Detection

The command automatically detects available LLM providers:

| Provider | How detected | Priority |
|---|---|---|
| `claude-cli` | `claude` command on PATH | 1 (highest) |
| `codex-cli` | `codex` command on PATH | 2 |
| `openai` | `OPENAI_API_KEY` in environment | 3 |
| `anthropic` | `ANTHROPIC_API_KEY` in environment | 4 |
| `grok` | `XAI_API_KEY` or `GROK_API_KEY` in environment | 5 |
| `deepseek` | `DEEPSEEK_API_KEY` in environment | 6 |

CLI tools are preferred over API keys because they handle their own authentication — no API key management needed.

### Generated agents.yaml

The first detected provider is assigned to the **architect**, the second to the **reviewer**. If only one provider is available, both agents use it.

```yaml
# Auto-generated by: agent-orchestra init
architect:
  provider: claude-cli
  model: claude-opus-4-6

reviewer:
  provider: codex-cli
  model: gpt-5.4
```

If more providers are available, they appear as commented suggestions for multi-reviewer setups.

### No providers found

If no CLI tools or API keys are detected, `init` prints setup instructions:

```
No providers available. Install a CLI tool or set an API key:
  Option 1: Install Claude CLI → https://claude.ai/download
  Option 2: Install Codex CLI → npm i -g @openai/codex
  Option 3: export OPENAI_API_KEY=sk-...
  Option 4: export ANTHROPIC_API_KEY=sk-ant-...
```

## Project Detection

| Signal | Detected Type |
|---|---|
| `package.json` or `tsconfig.json` | `node-ts` |
| `pyproject.toml` or `requirements.txt` | `python` |
| `Cargo.toml` | `rust` |
| No recognized markers | `generic` |

It also detects `tests/` and `docs/` directories to adjust superpower recommendations.

## Generated Files

### .agent-orchestra/agents.yaml

Per-agent provider configuration. Auto-generated based on detected providers and the current default model matrix:

- `claude-cli` -> `claude-opus-4-6`
- `codex-cli` -> `gpt-5.4`
- `openai` -> `gpt-5.4`
- `anthropic` -> `claude-sonnet-4-6`

Edit this file if you want a different provider/model pairing.

### AGENTS.md

Instructions for AI coding agents. Contains recommended superpowers and commands based on project type.

### .agent-orchestra/policy.yaml (with `--with-policy`)

Starter policy configuration with deny-by-default rules.

### .agent-orchestra/skills/* and .agent-orchestra/skillsets.builtin.yaml

Built-in review skills and bundled skillsets used by shipped superpowers like `plan-review` and `security-review`. These are generated automatically on every `ao init`, so built-in superpowers do not require extra manual skill setup.

### .agent-orchestra/skillsets.yaml (with `--with-skillsets`)

Starter custom skillset template with commented examples. This is separate from the auto-generated built-in skillsets file.

## Safe Behavior

- **agents.yaml exists and already matches:** skips
- **agents.yaml exists but differs:** `ao init` asks whether to refresh it
- **`--refresh-agents` flag:** replaces only `.agent-orchestra/agents.yaml` with newly detected defaults
- **AGENTS.md exists without Agent Orchestra content:** appends an Agent Orchestra section
- **AGENTS.md exists with Agent Orchestra content:** skips (no change)
- **`--force` flag:** overwrites all existing generated files

## Options

| Flag | Description | Default |
|---|---|---|
| `--path <path>` | Project root path | Current directory |
| `--yes` | Skip confirmation prompts | `false` |
| `--project-type <type>` | Override detected type | Auto-detect |
| `--with-policy` | Generate starter `policy.yaml` | `false` |
| `--with-skillsets` | Generate starter custom `skillsets.yaml` | `false` |
| `--refresh-agents` | Replace only `.agent-orchestra/agents.yaml` with newly detected defaults | `false` |
| `--force` | Overwrite existing generated files | `false` |

After init, run your first review:

```bash
ao run --target ./src --superpower security-review
ao run --target ./docs/plan.md --superpower plan-review --max-rounds 10 --auto-apply
```

When `--auto-apply` is enabled, Agent Orchestra patches the target file in place only for reviewer findings the architect explicitly acknowledged, and the reviewer re-reads that updated snapshot on the next round.
